# 0.4.1 Release Notes

`0.4.1` is the published structural-diagnostics and bounded-allocation follow-through release for `causal-order`.

This release keeps the synchronous raw-record ingress boundary from `0.4.0` intact while making translation failures safer to inspect, classify, control, and benchmark under pressure.

## Package Surface

Added:

* focused subpath exports for:
  * `causal-order/types`
  * `causal-order/clock`
  * `causal-order/compare`
  * `causal-order/validate`
  * `causal-order/anomalies`
  * `causal-order/order`
  * `causal-order/translate`
* explicit nested translation diagnostics on `TranslationAnomaly`
* stable translation `classification` and `fieldReference` metadata
* explicit translation strictness-policy controls through:
  * `record_failure`
  * `optional_field_failure`
  * exported `TranslateBatchPolicyError`
* deterministic translation diagnostic ordering metadata:
  * `ordering.kind`
  * `ordering.sequence`
  * `ordering.recordIndex`
  * `ordering.fieldOrder`

Finalized:

* the split between structural and mapper-surface translation failures as machine-readable contract surface
* stable translation anomaly categories for downstream tooling
* deterministic emission order for translation anomalies with identical input
* additive focused-import entrypoints without breaking the existing top-level package contract

## Diagnostic Contract

`0.4.1` makes the translation anomaly surface safer to build against directly.

That now includes:

* nested `diagnostic` objects for mapping and structural failures
* stable `classification` metadata instead of text-only interpretation
* stable `fieldReference` metadata for ingress-facing field attribution
* explicit `policy` metadata so strictness decisions are inspectable
* stable emitted ordering metadata for testing, snapshotting, and downstream tooling

This release is not a terminal-formatting pass.
It is a machine-readable diagnostics release.

## Allocation Discipline

`0.4.1` also tightens anomaly-heavy allocation behavior without changing the public batch-ordering contract.

That includes:

* shared internal anomaly collection so `orderEvents()` no longer retains a full validation-record array during anomaly detection
* lazy graph-evidence allocation in the batch ordering path
* reduced intermediate ordering structures in `orderValidatedEvents()`
* lower allocation churn in same-node sequence handling

The result is a materially cheaper anomaly-heavy batch path under the repo's `250k` GC-observed validation harness, while keeping the same ordering and anomaly semantics.
