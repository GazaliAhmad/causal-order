# 0.5.0 Release Notes

`0.5.0` is the published stability-and-contract-design release for `causal-order`.

This release does not widen the package into payload-aware domain resolution.
It clarifies what the project intends to preserve, what names new code should prefer, and which semantic areas belong in extension hooks rather than in the payload-agnostic core runtime.

## What Landed

`0.5.0` now adds:

* an exported-surface inventory for the current package-facing API
* explicit API-clarity decisions for:
  * `compareByHlc()` versus `compareClocks()`
  * `compareDeterministically()` versus `compareWithTieBreaker()`
  * `orderValidatedEvents()` and its current `internal` coordination parameter
* a default-behavior compatibility inventory and matching decision record
* domain-semantic design notes for:
  * contradictory events
  * entity forks
  * semantic dedupe across different identifiers
* explicit core-versus-extension boundary decisions for contradiction handling, fork handling, and semantic dedupe
* migration notes for the preserved surface
* targeted stability-surface tests that lock in:
  * compatibility helper aliases and alias parity
  * warning-visible severity governance
  * the rule that less emitted telemetry does not create more causal certainty

## Public Naming Direction

`0.5.0` now makes the preferred helper names explicit:

* prefer `compareByHlc()` for direct HLC comparison
* prefer `compareDeterministically()` for deterministic fallback comparison

Compatibility posture:

* `compareClocks()` still works as compatibility surface
* `compareWithTieBreaker()` still works as compatibility surface
* the first migration signal is deprecation guidance rather than abrupt removal

## Preserved Behavior Direction

This release also makes the current warning-visible defaults explicit:

* `strict: false` remains the default
* `detectAnomalies: true` remains the default
* unresolved ordering remains visible in non-strict mode rather than being hidden by default
* translation optional-field handling remains warning-visible by default through `optionalFieldFailure: "warn"`

That means `0.5.0` preserves the project’s core honesty rule:

* anomaly visibility is part of the ordinary contract
* unresolved output should not be normalized away silently
* reducing diagnostic output does not make ordering results more justified

## Core Boundary Direction

`0.5.0` does not turn the payload-agnostic core into a domain-resolution engine.

The release makes these boundaries explicit:

* contradiction detection may belong in core, but contradiction resolution belongs in policy
* fork detection may belong in core if it stays payload-agnostic and caller-grouped, but branch choice belongs in policy
* semantic dedupe candidate reporting may belong in core, but suppression, collapse, or rewrite decisions belong in policy

The core package should therefore not claim that it:

* resolves contradictory business records automatically
* chooses canonical entity truth automatically
* deduplicates business objects correctly by itself

The repo now describes `0.5.0` as the published release line rather than only as an active milestone.

That includes alignment across:
