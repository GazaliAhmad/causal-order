# 0.8.0 Release Notes

`0.8.0` is the first adoption-maturity release for `causal-order`.

It builds on the published `0.7.0` transferability baseline and turns that baseline into a clearer package-evaluation and maintainer-adoption path.
The release keeps the runtime claim surface narrow while making the public docs flow read more like one deliberate product story.

## What Landed

`0.8.0` now adds:

* a stronger first-run adoption flow across the public package surfaces:
  * `README.md`
  * `guides/README.md`
  * the website landing and docs navigation
* a clearer evaluator path for:
  * when to use the package
  * when not to use it
  * which examples and guides to read first
* a clearer maintainer handoff path for:
  * release process
  * compatibility posture
  * support and maintenance expectations
* more deliberate release-facing doc alignment:
  * changelog wording
  * release-history links
  * docs-sync coverage for the package-facing path
* a first operator-maturity follow-through across the practical workflow docs:
  * replay inspection
  * streaming reconciliation
  * anomaly interpretation
  * incident-style review

## Adoption-Flow Direction

`0.8.0` is about helping technically strong readers judge the package quickly and accurately.

That means:

* an evaluator can understand the package value and boundary from the public docs flow alone
* a maintainer can find the practical starting points for release, compatibility, and support posture without reconstruction work
* a new engineer can move from README to guides, examples, and API docs without guessing which surface is canonical

The release is therefore about adoption clarity, not about broadening semantics.

## Release Boundary

`0.8.0` does not widen the core runtime contract.

It does not add:

* new causal-ordering semantics
* new translation semantics
* new streaming guarantees
* official adapters, connectors, or deployment-specific integrations
* payload-aware contradiction, fork, or semantic-dedupe resolution inside the payload-agnostic core

The release is about clearer adoption, maintenance, and evaluation posture rather than runtime expansion.

## Operator-Maturity Follow-Through

`0.8.0` also begins the next `0.8.x` follow-through around operational review.

That means the docs now do more than explain package mechanics.
They also help operators answer:

* what should we inspect first during bounded replay review?
* what does correction-capable stream output mean operationally?
* what do common anomaly types usually signal?
* how should incident review separate source evidence from derived explanation?

This is still documentation and workflow maturity, not a new runtime semantics line.

## Release Result

The `0.8.0` release line now reads consistently across:

* `README.md`
* `CHANGELOG.md`
* `ROADMAP.md`
* `docs/releases/0.8.0.md`
* `guides/README.md`
* `MAINTENANCE.md`
* `RELEASE_PROCESS.md`
* `COMPATIBILITY.md`
* the website guides, wiki, and API surfaces

At this stopping point, `0.8.0` is the first release where the package's evaluator flow, maintainer handoff, operator workflow guidance, and release-facing docs all read like one coherent product line.
