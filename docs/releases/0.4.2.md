# 0.4.2 Release Notes

`0.4.2` is the developer-experience follow-through release for the current public ingress and diagnostics surface in `causal-order`.

This release does not widen the core runtime API.
It makes the existing package surface easier to evaluate, learn, and verify through runnable examples, package-facing guidance, and docs synchronization enforcement.

## What Landed

`0.4.2` now adds:

* runnable ingress examples that show the real `translateBatch()` to `orderEvents()` path
* package-facing examples that import from `causal-order` or published public subpaths instead of repo-internal `../dist/...` paths
* quick-start scenario guidance for:
  * audit timeline reconstruction
  * replay pipelines
  * distributed debugging
  * offline sync inspection
* package-facing policy guidance for:
  * `strict: true` versus `strict: false`
  * `lateArrivalPolicy: "flag"`
  * `lateArrivalPolicy: "drop"`
  * `lateArrivalPolicy: "emit_correction"`
  * `lateArrivalPolicy: "fail"`
* docs synchronization enforcement through:
  * a dedicated docs-sync check
  * release verification wiring
  * CI coverage for docs-only and example-facing changes

## Example Posture

The examples are now intentionally written from the customer point of view.

That means:

* runnable examples use the public package surface
* copied example code still looks like the right starting point in a consumer project
* the ingress contract is shown directly instead of being hidden behind repo-local wrappers

The current runnable ingress entry points are:

* `examples/ingress-minimal.mjs`
* `examples/ingress-replay-pipeline.mjs`

The broader runnable example set now follows the same public-import rule.

## Guidance Layer

`0.4.2` also turns the examples into a clearer package-facing onboarding path.

That includes:

* a quick-start scenario guide that routes readers by workload shape
* direct links from the README, guides index, and wiki entry points to the runnable examples
* a dedicated policy guide for strictness and late-arrival decisions

The goal is not broader abstraction.
The goal is making the current package honest and teachable on its own terms.

## Synchronization Enforcement

This release is not only wording.
The repo now verifies that the package-facing docs and runnable examples stay aligned.

That includes checks that:

* the key README, guides, and wiki entry points still reference the current runnable examples
* runnable examples do not regress back to repo-internal import paths
* the examples still run as part of the docs synchronization gate

## Release Boundary

`0.4.2` remains inside the same core-package boundary established by `0.4.0` and `0.4.1`.

It does not add:

* new runtime entrypoints or subpaths
* async ingress APIs
* file-format adapters
* companion CLI layers
* environment-aware orchestration helpers

The release is about package-facing teachability and verification, not about widening the API.
