## `Must decide`

### 1. Root-Level Compatibility Aliases

Status:

* `completed`

Relevant evidence:

* `src/compare/hlcCompare.ts`
* `src/order/tieBreakers.ts`
* `test/unit/subpath-exports.test.mjs`
* `test/unit/stability-candidate-surface.test.mjs`

Current state:

* `compareWithTieBreaker()` has now been removed from the public package surface
* `compareClocks()` remains as deprecated root-only compatibility surface
* the focused subpaths still exclude both compatibility aliases

Current decision:

* keep `compareClocks()` as deprecated root-only compatibility surface through `0.9.x`
* remove `compareClocks()` at `1.0.0`
* `compareWithTieBreaker()` has already been removed before `1.0.0`

Decision rationale:

* `compareByHlc()` is already the clearer long-term public name for direct HLC comparison
* `compareClocks()` still has some migration value because it was the earlier ergonomic public name
* `compareWithTieBreaker()` was the weaker retention candidate because it overlapped directly with the same deterministic fallback job now better expressed by `compareDeterministically()`

What remains:

* release-facing follow-through so the eventual `1.0.0` removal does not arrive as a surprise

### 2. `applyTieBreaker()` Classification

Status:

* `partially satisfied`

Relevant evidence:

* `src/compare.ts`
* `guides/stability/decision-record-api-clarity-0.5.0.md`

Current state:

* `applyTieBreaker()` remains public
* the source now documents it explicitly as a low-level deterministic primitive
* the published-surface and subpath tests now treat it as intentional public surface

Current decision:

* keep `applyTieBreaker()` public into `1.0.0`
* classify it explicitly as an intentional low-level deterministic primitive
* keep `compareDeterministically()` as the primary user-facing fallback helper

Decision rationale:

* `compareWithTieBreaker()` and `compareDeterministically()` overlap at the same public level
* `applyTieBreaker()` serves a different role as the lower-level primitive underneath that higher-level deterministic fallback story
* keeping it public is reasonable as long as the docs clearly rank it below the primary helper path

What remains open:

* whether any broader package-facing docs should say more than the current source and test posture already does

### 3. `orderValidatedEvents()` Advanced Signature

Status:

* `partially satisfied`

Relevant evidence:

* `src/order/orderEvents.ts`
* `src/order/internalOrderValidatedEvents.ts`

Current state:

* the public `orderValidatedEvents()` signature is now narrowed to the validated-events plus options path
* the older repo-coordination `internal` bag is no longer part of the public function signature
* internal coordination still exists privately for repo-owned callers such as:
  * `orderEvents()`
  * `orderEventStream()`

Current decision direction:

* keep `orderValidatedEvents()` as a public concept
* do not preserve the old `internal` coordination parameter as stable public contract
* keep internal coordination plumbing behind repo-local support paths only

Decision rationale:

* ordering already-validated events remains a real public need
* the old `internal` bag exposed implementation coordination more than domain contract
* narrowing the public function while preserving internal support keeps the public API cleaner without dropping repo-owned efficiency paths abruptly

What remains open:

* whether any release-facing migration wording should call out the narrowed public shape more explicitly before `1.0.0`

### 4. `StreamOrderBatch` Final Type Name

Status:

* `completed`

Relevant evidence:

* `src/types.ts`
* `guides/stability/exported-surface-inventory-0.5.0.md`

Decision rationale:

* `0.9.0-a` resolves the old naming ambiguity with a hard rename from `OrderBatch` to `StreamOrderBatch`
* the new name matches the actual contract: one emitted `orderEventStream()` batch
* this is type-surface churn, but it is much cheaper before `1.0.0` than after the public contract freezes
