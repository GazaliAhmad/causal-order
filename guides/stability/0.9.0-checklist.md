# `0.9.0` Checklist

The roadmap already defines `0.9.x` as the final stabilization line before `1.0.0`.
That means the job here is to resolve ambiguity, make final compatibility decisions,
do a last pass on docs, examples, migration notes, operator guidance, and release
packaging, and make sure the package does not overclaim beyond the implemented
contract.

This note turns that into a concrete checklist from the current repo state.

## Core `0.9.0` Checklist

### 1. Decide The Final Fate Of Root-Level Compatibility Aliases

`compareClocks()` and `compareWithTieBreaker()` are still exported from the root
surface and explicitly marked deprecated in the source:

* `src/compare/hlcCompare.ts`
* `src/order/tieBreakers.ts`

The focused subpaths already exclude them, and the tests lock that in:

* `test/unit/subpath-exports.test.mjs`
* `test/unit/stability-candidate-surface.test.mjs`

`0.9.0` should make the explicit `1.0.0` call:

* keep both
* remove both
* keep one and drop one

### 2. Classify `applyTieBreaker()`

The older stability decision record already identified this as a real review item,
and that still looks true in the live public surface:

* `guides/stability/decision-record-api-clarity-0.5.0.md`
* `src/compare.ts`

The public story is already clear for `compareDeterministically()`.
It is still softer for `applyTieBreaker()`.

`0.9.0` should decide whether `applyTieBreaker()` is:

* an intentional public low-level API
* a compatibility holdover
* a pre-`1.0.0` cleanup candidate

### 3. Narrow Or Formally Bless The `orderValidatedEvents()` Advanced Signature

The code still states that the third `internal` parameter should not be treated as
stable long-term public contract:

* `src/order/orderEvents.ts`

The helper type is intentionally kept off the public export surface so it can still
change:

* `src/order/internalOrderValidatedEvents.ts`

This is a real `0.9.0` blocker.
Before `1.0.0`, the project should either:

* replace the current advanced shape with a stable public advanced API
* narrow the signature
* or remove the ambiguity explicitly and document the surviving contract clearly

### 4. Decide Whether `OrderBatch` Is The Final Stream Result Type Name

The type is still named `OrderBatch` even though it is stream-specific and carries:

* `watermark`
* `anomalyHorizon`
* `correction`
* `isFinal`

Source:

* `src/types.ts`

The earlier exported-surface audit explicitly flagged this as a soft naming area:

* `guides/stability/exported-surface-inventory-0.5.0.md`

`0.9.0` should either:

* keep `OrderBatch` intentionally as the long-term name
* or rename it before `1.0.0`

### 5. Freeze The Wording And Semantics Of `strict`, `detectAnomalies`, And `allowUnknownOrder`

The behavior is already coherent and covered by tests, but the documentation still
frames parts of this as explicit review surface rather than fully settled
long-term wording:

* `guides/stability/default-behavior-compatibility-inventory-0.5.0.md`
* `test/unit/stability-candidate-surface.test.mjs`

Recommendation:

* prefer docs clarification first
* avoid renaming unless the package-facing wording still feels genuinely confusing

### 6. Decide What To Do With The Exported "Draft" Extension-Policy Interfaces

Several interfaces are public in `src/types.ts`, but are still described more as
boundary direction than as a finished extension framework:

* `PolicyVisibilityRecord`
* `CausalContradictionPolicy`
* `ForkResolutionPolicy`
* `SemanticDedupePolicy`

Relevant sources:

* `src/types.ts`
* `guides/extension-boundary-guide.md`

`0.9.0` should choose one of two paths:

* preserve them intentionally as part of the stable boundary surface
* mark them more explicitly as pre-`1.0.0` provisional, or move them out of the stable claim surface

### 7. Lock The Root-Versus-Subpath Story Everywhere

The README and upgrade guide are already aligned that:

* documented primary names are the preferred path
* compatibility aliases may still exist at the root
* focused subpaths are the tighter long-term public path

Relevant sources:

* `README.md`
* `guides/upgrade-expectations.md`

`0.9.0` should make sure every example, guide, and migration note follows that same
story consistently.

### 8. Add Final Public-Surface Tests For The `1.0.0` Decisions

The repo already has stability-oriented and subpath-export tests:

* `test/run.mjs`
* `test/unit/stability-candidate-surface.test.mjs`
* `test/unit/subpath-exports.test.mjs`

`0.9.0` should extend these to lock in whichever final alias, naming, and contract
decisions are chosen, rather than leaving those decisions in prose only.

### 9. Ship A Release-Surface Pass, Not A Capability Pass

`0.9.0` should be about aligning the public release surface:

* `CHANGELOG.md`
* `README.md`
* `ROADMAP.md`
* release notes
* migration wording
* package metadata

It should not be the place for:

* new incident analog families
* new policy runtime
* ecosystem expansion
* governance-heavy additions

## Real `0.9.0` Blockers

These look like the genuine blockers before `1.0.0`:

* root alias survival or removal decision
* `applyTieBreaker()` classification
* `orderValidatedEvents()` signature decision
* `OrderBatch` keep-or-rename decision
* exported draft policy-interface stance

## Recommended `0.9.0` Scope

The cleanest `0.9.0` scope is:

1. API and naming finalization
2. compatibility decision line
3. docs, examples, migration, and release alignment
4. public-surface test locking

That keeps `0.9.0` as the right bridge to `1.0.0` instead of turning it into
another maturity line that quietly reopens scope.
