The purpose is to separate:

It is a repo-local planning artifact for the final pre-`1.0.0` stabilization line.
It is not intended as public package-facing guidance.

## `Must decide`

### 1. Exported Extension-Policy Interface Stance

Status:

* `resolved`
* `landed`

Relevant evidence:

* `src/types.ts`
* `guides/extension-boundary-guide.md`
* `guides/supported-vs-unsupported-usage.md`
* `website/src/generated/api.json`

Previous state:

* `PolicyVisibilityRecord`, `CausalContradictionPolicy`, `ForkResolutionPolicy`, and `SemanticDedupePolicy` are publicly exported from `src/types.ts`
* the source and guides still describe them as draft or boundary-direction surface
* the website API already surfaces them publicly under an extension-policy section

Why this was a real blocker:

* if they remain exported and surfaced, callers can reasonably read them as intentional public contract
* if they are called draft forever, they become "draft by accident" rather than deliberate boundary vocabulary
* if they are treated as stable without clearer wording, the package risks overclaiming a policy framework it does not actually implement

Decision taken:

* preserve them as intentional public boundary interfaces
* describe them as boundary vocabulary for higher-layer policy, not as a built-in extension runtime or finished plug-in framework
* replace "draft by implication" wording with a sharper statement of what is and is not stable about them

Decision rationale:

* the repo has already spent real design energy defining this boundary
* the names are already part of the public type and website API surface
* the healthier freeze line is not pretending they do not exist, but making their role explicit:
  * core can surface candidates and operator-visible records
  * higher layers own contradiction action, fork action, and semantic dedupe action

What remains open:

* whether any residual public wording still says `draft`
* whether the API docs should present them as a normal type group or as explicitly boundary-only types

### 2. `strict`, `detectAnomalies`, And `allowUnknownOrder` Wording Freeze

Status:

* `resolved`
* `landed`

Relevant evidence:

* `README.md`
* `guides/policy-guidance.md`
* `guides/stability/default-behavior-compatibility-inventory-0.5.0.md`
* `guides/stability/decision-record-default-behavior-0.5.0.md`
* `test/unit/stability-candidate-surface.test.mjs`
* `website/src/generated/api.json`

Current state:

* the runtime semantics are already well-covered by tests
* README and policy guidance already explain the important boundary rules:
  * `strict` is the main fail-fast switch for ordering and validation, not translation policy
  * `allowUnknownOrder: false` strengthens severity posture without inventing stronger certainty
  * `detectAnomalies: false` reduces emitted diagnostic output rather than making the result truer or cleaner

Why this is close to done:

* there is no obvious runtime-behavior ambiguity left
* the main remaining work is consistency, not renaming
* the website API descriptions already lean in the same direction

Current recommended direction:

* preserve all three names as-is into `1.0.0`
* treat the work here as wording freeze rather than option redesign
* make sure every public explanation uses the same three ideas:
  * fail-fast
  * uncertainty visibility
  * diagnostic-output shaping

What remains open:

* whether any residual public wording still drifts from the same three-part mental model
* whether any first-time-reader surfaces still overread `detectAnomalies` as a correctness mode

### 3. Core-Versus-Policy Overclaim Risk

Status:

* `partially satisfied`

Relevant evidence:

* `guides/extension-boundary-guide.md`
* `guides/supported-vs-unsupported-usage.md`
* `README.md`
* `guides/stability/decision-record-core-boundaries-0.5.0.md`

Current state:

* the public guides already say the core owns payload-agnostic detection, ordering, and visibility
* the same guides already say higher layers own contradiction handling, fork handling, semantic dedupe action, and environment-specific glue
* README also frames the package as a deployable ordering engine rather than a complete event platform

Where the remaining softness is:

* the source and guides are more careful than the mere existence of the exported interfaces might suggest
* this creates a small but real mismatch:
  * docs say "boundary direction"
  * public types and API site say "these are here right now"

Current recommended direction:

* keep the core-versus-policy split exactly where it is
* tighten wording so the exported interfaces reinforce that split instead of blurring it
